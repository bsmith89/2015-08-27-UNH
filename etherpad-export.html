#### Commands we've seen ####

  • whoami   - print username
  • echo        - print all of the 'arguments' (words that come after the command)
  • cd            - change to the given directory (without arguments, changes to the "home" directory)
  • pwd         - Print working directory, print the directory that you are currently "in" (last changed to)
  • ls             - (lowercase L, lowercase S) list the files in the working directory
  • clear        - remove text from the terminal screen (doesn't change anything else)
  • mkdir       - make a new directory
  • touch [file]       - If file doesn't exist, make the file (empty)
  • cat [file]           - print the contents of a file to the screen
  • rm [file]           - remove the file (***THIS CAN BE VERY DANGEROUS. IT IS COMPLETELY GONE, NOT JUST IN YOUR TRASH CAN***)
  • cp [path1] [path2]  - Copy path1 to path2. Use -r to copy directories recursively.
  • wc           - word count
  • sort       - Print sorted content of file(s) 
    • -n sorts numerically
    • -r reverses order
  • head     - Print the first 10 lines of a file
    • `head -1` prints the first line, `head -5` prints the first 5 lines, etc.
  • uniq      - Print contents; filter duplicate lines from the input
    • Possible to redirect results into a file of the same name?

#### Getting Help for Commands ####

  • On Windows/Mac/Linux:
    • Google '[command] manual'
    • Type `[command] --help` or `[command] -h` (`ls` is an example of a command where this doesn't work)
  • On UNIX systems (Mac and Linux)
    • Type `man [command]` in the terminal

#### Paths ####

  • The "location" of a file
  • Forward slash `/` at the beginning of the path (or as the entire path) means "root"
  • A complete path is a path that starts at the root
  • A relative path is relative to the current working directory; does not start with a slash
  • `..` stands for the parent directory.  `cd ..` will move you one directory "up"
  • `.` means the current working directory `cd .` will not move you anywhere
  • `*` (astrisk) means 'wildcard'...or rather, anything in a file name that exists
    • e.g. `*.pdb` as an argument really means, "all of the files that end with `.pdb`"

#### Working with Output ####

  • `>` redirects the output of a command to a file (WRITES OVER WHATEVER IS THERE)
    • e.g. `wc -l *.pdb > len.txt` sends the line count of each file that ends with `.pdb` to the file lens.txt
    • `cat len.txt` will then print that file's contents to the screen
  • `>>` appends the output to the file (rather than overwriting)
  • `|` "pipes" the output of one command into the next command

#### Editing text files on Windows ####
  • nano is broken with Git Bash
  • Use `notepad [FILENAME]`
  • Usually helpful to leave a blank line at the end of the file (equivalent to `\r` at the end of the last line)
  • If you're having file ending issues on Windows, try the command `dos2unix` like so:
    • dos2unix [filename] | sort | uniq
  • Let me know in the chat if that works or doesn't work (I can't test it on my mac)

#### Bash scripts ####
  • Text files that usually end in `*.sh`
  • Run with `bash [scriptname]`, e.g. `bash backup.sh`
  • Comment lines start with `#`



#### Challenge Question #1 ####
Why does uniq only remove adjacent duplicates?
The command uniq removes adjacent duplicated lines from its input. For example, if a file salmon.txt contains:

coho
coho
steelhead
coho
steelhead
steelhead

then uniq salmon.txt produces:

coho
steelhead
coho
steelhead

1.  Why do you think uniq only removes adjacent duplicated lines? (Hint: think about very large data sets.)
2.  What other command could you combine with it in a pipe to remove all duplicated lines?

#### IPython Notebook tips ####
  • Pressing shift+enter executes a cell and moves to next

#### Slicing ####
  • Last index is not included when slicing, e.g. `some_list[:-1]` contains all items of `some_list` except the last
  • iterate backwards across entire list by 1:    `some_list[::-1]`

#### Using the Conda package manager ####
  • `conda search [packagename]`, e.g. `conda search seaborn`     - search for a package to install
  • `conda install [packagename]`      - install a package with conda
  • `conda update [packagename]`     - update a package

Find packages that do useful things using Google, GitHub, etc. If they can't be installed with conda, you can usually use `pip install [packagename]`.

##### Configuring Git #####

`git config --global user.name "<your name here>"`
`git config --global user.email "<your email here>"`ls

`git config --global color.ui "auto"`
`git config --global core.editor "nano -w"
`git config --list`         # Print configured settings
`git config --list --global`     # Print global settings
`git config --list --global | grep user`     # see only the global git settings with "user" in them

git config --global alias.l 'log --pretty=oneline -n 20 --graph --abbrev-commit'
^^ dot lower case L (not one(1))


##### Using Git #####

`git init`    # when inside a directory, tell git that you want to consider that directory a repository
`git status`    # tell me what's up
`git add`      # with file name arguments, adds those files to the index (ready to be saved)
    `git add -A`    # Add all changed and/or untracked files in the current directory or below
`git commit -m "<commit message>"`    # Save the current state of the index (also known as the staging area) as a version.  Use the `-m` flag followed by a double-quote surrounded sentence saying what you've changed and the purpose of the change.
`git log`
* to print a git log that is easier to read and has abbreviated commit IDs you can use:
`git log --pretty=oneline --abbrev-commit`
'git remote set-url origin linkFromGithub'       - reset the URL for the local repository pointing to the GitHub remote repo named "origin"


##### Git tips on Windows #####

You may get a message like:
"warning: LF will be replaced by CRLF in ingredients.txt.
The file will have its original line endings in your working directory."

This is perfectly normal, and will be beneficial if collaborating with others using Mac or Linux  (systems that use different line ending characters). It is recommended you run

    `git config --global core.autocrlf true`

See https://help.github.com/articles/dealing-with-line-endings/ for more details.
